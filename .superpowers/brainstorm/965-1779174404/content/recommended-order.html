推荐顺序

按依赖关系和效率最优排列

第一步：产品详情页 UI 重构

核心转化页面，先把它设计好。设计时就考虑移动端布局，避免后面返工。

理由

  • 产品页是用户决策的关键页面
  • AGENTS.md 中明确延后到现在
  • 先设计好再适配，比分两次做好

风险

  • 改动最大的单页
  • 需要视觉设计决策
第二步：响应式适配

基于新的产品详情页，全站做移动端适配。产品页不用重复做两次。

理由

  • 产品页已在第一步设计好，直接适配
  • 其他页面（购物车/结算/个人中心/后台）统一处理

风险

  • 涉及页面最多
  • Tailwind 4 的响应式方案需要确认
第三步：SEO + middleware 迁移 + 部署（可并行）

三个独立小任务，互不依赖。部署放最后——等前面的东西就绪再上线。

🔍
SEO

metadata / og / sitemap

🔧
proxy 迁移

middleware → proxy.ts

🚀
Vercel 部署

环境变量 + 上线

这个顺序的核心逻辑

产品详情页 → 响应式 → SEO+迁移+部署。

为什么不是响应式优先？因为产品详情页是唯一需要"重设计"（不光是适配）的页面。如果先做响应式，产品详情页的适配工作在 UI 重构后会被覆盖掉，等于做两遍。

为什么 SEO、proxy 迁移、部署放最后？因为这三个互不依赖，可以灵活穿插。部署放最后是自然的——前面的东西做好再上线。
