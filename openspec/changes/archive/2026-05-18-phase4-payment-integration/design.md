## Context

当前系统完成到 Phase 3（购物车 + 下单），`POST /api/orders` 创建订单后直接显示"支付功能即将上线"。`PaymentRecord` 表已在 schema 中定义但从未写入。项目暂未取得企业营业执照等支付资质，无法接入真实微信/支付宝。

本阶段目标是：构建完整的支付基础设施，用模拟支付跑通全链路。将来取得资质后，只需替换支付网关层（模拟 → 真实 SDK），上层订单流程和 UI 不变。

## Goals / Non-Goals

**Goals:**
- 下单后跳转支付页面，用户选择支付方式并完成（模拟）支付
- 订单状态机支持完整生命周期：`pending_payment` → `paid` / `expired` / `cancelled`
- 下单事务同步创建 PaymentRecord
- 订单 30 分钟超时自动取消 + 手动取消 + 库存回滚
- 订单列表展示真实数据，支持"去支付""取消""查看详情"操作
- 订单详情页展示完整订单信息

**Non-Goals:**
- 真实微信/支付宝 SDK 接入
- 支付回调/通知处理（模拟支付不产生回调）
- 退款功能
- 后台管理中的订单管理（属于 Phase 5）
- 数据库 schema 变更（现有表结构足够）

## Decisions

### 1. 模拟支付：即时确认无延迟

模拟支付 API (`POST /api/payment/pay`) 收到请求后立即标记订单为已支付，不模拟延迟、不模拟失败。前端点击"确认支付"→ API 返回成功 → 跳转成功页。简单直接，将来替换为真实支付时改为异步回调模式。

**替代方案考虑：**
- 模拟支付延迟 + 可能失败：增加复杂度但无实际收益，mock 只是为了跑通流程
- 直接在前端标记支付成功：跳过 API 层，未来替换成本高

### 2. 下单即扣库存

保留当前行为：`POST /api/orders` 事务中扣减库存。不等到支付完成再扣。

**理由：** 茶叶为小众精品，SKU 库存量小（每款几件到几十件），超卖风险高于库存锁定风险。搭配 30 分钟超时取消 + 库存回滚机制，平衡可用性。

**替代方案考虑：**
- 支付后扣库存：无超卖风险，但实现复杂（需处理付款时库存不足的边际情况），不适合低库存商品
- 预扣 + 定时释放：需要定时任务基础设施，当前 Vercel 免费层 Cron 可用但引入额外依赖

### 3. 支付方式选择放在支付页而非结算页

结算页只负责选中地址 + 提交订单，不选支付方式。订单创建后跳转 `/payment/[orderId]`，在该页选择微信/支付宝并支付。

**理由：** 贴近真实电商流程（淘宝/京东均为此模式）。支付方式是支付动作的一部分，不应与地址选择耦合。将来对接真实支付时，支付页可进一步分化为微信 H5 / 支付宝收银台。

### 4. 超时处理：前端倒计时 + 手动取消 + 服务端兜底

**前端：** 支付页展示 30 分钟倒计时，倒计时归零自动调用取消 API。
**服务端：** 取消 API 检查订单是否仍为 `pending_payment` 才执行取消。在订单查询（订单列表、订单详情）时额外检查超时，对超时的 `pending_payment` 订单自动触发取消。

**理由：** 不需要定时任务或 Cron Jobs，利用用户访问触发清理。简单可靠，适合当前阶段。

**替代方案考虑：**
- Vercel Cron Jobs 定时扫描：更可靠但引入额外依赖，免费层仅 2 个 Cron 任务
- 不做超时处理：库存可能被永久锁定

### 5. 订单状态值

```
pending_payment  ← 下单时设置
paid             ← 模拟支付成功时设置
expired          ← 30 分钟超时自动取消
cancelled        ← 用户手动取消
```

现有订单的 `pending` 值保留不变（兼容已有数据），新订单全部使用 `pending_payment`。

### 6. 不新增数据库表

`PaymentRecord` 表已存在且字段完备（`orderId`, `method`, `amount`, `status`, `thirdPartyId`）。模拟支付不产生真实第三方 ID，`thirdPartyId` 留空，将来对接真实支付时填写。

## Risks / Trade-offs

- **[Mock → Real 切换风险]** 模拟支付的即时同步模式与真实支付的异步回调模式不同 → 替换时需修改支付 API 和支付页的等待逻辑。缓解：支付 API 接口保持 `POST /api/payment/pay`，返回值结构预留扩展字段
- **[库存锁定]** 下单即扣库存，用户不付款会导致 30 分钟内库存不可售 → 缓解：茶叶非高频快消品，30 分钟窗口可接受；30 分钟后自动回滚
- **[浏览器关闭后无主动取消]** 用户关闭支付页后无法手动取消 → 缓解：订单列表和详情页的查询触发过期检测，用户下次访问时自动清理
